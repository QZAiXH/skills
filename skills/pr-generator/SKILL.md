# PR Generator

自动分析 git 分支变更，生成结构化的 PR 标题和描述。

## ⚠️ 核心约束

**PR 描述必须且只能包含"## 主要变更"这一个章节，不得包含任何其他内容。**

这是用户的明确要求，违反此规则视为任务失败。在开始工作前，请先阅读 `references/output-examples.md` 了解正确和错误的输出格式。

## 何时使用

- 用户明确提到"PR"、"pull request"、"合并"
- 用户说"提 PR 到 test/main"、"生成 PR 信息"
- 用户完成功能开发，准备代码审查
- 用户需要总结当前分支的改动

## 工作流程

### 1. 确定目标分支

如果用户明确指定目标分支（如"合并到 test"），使用该分支。否则询问用户目标分支是什么。

常见目标分支：
- `test` - 测试环境分支
- `main` / `master` - 生产环境主分支
- `develop` - 开发分支
- 其他自定义分支

### 2. 收集变更信息

并行执行以下 git 命令收集完整的变更上下文：

```bash
# 当前分支信息
git branch --show-current

# 相对于目标分支的提交历史
git log <target-branch>..HEAD --oneline

# 文件变更统计
git diff <target-branch>...HEAD --stat

# 关键文件的详细变更（选择性查看）
git diff <target-branch>...HEAD -- <关键文件路径>
```

**重要**：使用三点语法 `...` 来对比分支分叉点以来的变更，这样能准确反映当前分支引入的改动。

### 3. 分析变更内容

基于收集的信息，识别：

**功能模块**：
- 从文件路径识别涉及的模块（如 `internal/service/notification/` → 通知模块）
- 从 proto 文件识别新增的 API 接口
- 从 schema 文件识别数据模型变更

**变更类型**：
- 新功能开发（feat）：新增文件、新增接口、新增数据表
- Bug 修复（fix）：修复逻辑、错误处理、判空处理
- 重构（refactor）：代码结构调整、性能优化
- 文档（docs）：README、注释更新
- 其他（chore）：配置变更、依赖更新

**变更规模**：
- 单一模块 vs 多模块
- 小改动（<100 行）vs 大改动（>1000 行）

### 4. 生成 PR 标题

**格式规则**：
```
<type>: <简短描述>
```

**type 选择逻辑**：
- 如果包含多种类型的变更，选择最主要的类型
- 新功能 + Bug 修复 → 优先 `feat`
- 多个独立功能 → 使用 `feat` 并在描述中体现"与"

**标题要求**：
- 长度控制在 50 字符以内
- 使用动宾结构：`实现 XX 功能`、`优化 XX 模块`、`修复 XX 问题`
- 体现核心价值，不罗列细节

**示例**：
- ✅ `feat: 消息通知系统与市场观点优化`
- ✅ `fix: 用户认证模块空指针异常`
- ✅ `refactor: 重构权限系统数据库连接`
- ❌ `feat: 添加了通知列表接口并且优化了市场观点的判空还有...`（太长）
- ❌ `更新代码`（不明确）

### 5. 生成 PR 描述

**在生成描述前，必须先阅读 `references/output-examples.md` 了解正确格式。**

**CRITICAL：PR 描述必须且只能包含"主要变更"部分，不得包含任何其他章节。**

**严格的输出结构**：

```markdown
## 主要变更

### 1. <模块名称>
- **<子功能点>**：<具体说明>
  - <细节 1>
  - <细节 2>
- **<子功能点>**：<具体说明>

### 2. <模块名称>
- <变更说明>
```

**这是完整的 PR 描述。描述从 `## 主要变更` 开始，到最后一个变更点结束，不要添加任何其他内容。**

**内容组织原则**：

1. **按模块分组**：不同功能模块使用三级标题（### 1. 模块名）
2. **突出重点**：API 接口、数据模型、核心逻辑优先说明
3. **层次清晰**：使用加粗、列表、缩进体现层次关系
4. **具体明确**：
   - ✅ "新增消息列表查询接口 `GET /api/v1/notification`"
   - ❌ "添加了一些接口"
5. **技术细节适度**：
   - 包含：接口路径、数据表名、关键字段、支持的功能
   - 不包含：具体实现方式、代码行数、文件路径

**绝对禁止的内容**（违反此规则视为失败）：

- ❌ 任何"主要变更"之外的章节标题，包括但不限于：
  - "功能概述"、"技术实现"、"技术细节"、"技术收益"
  - "测试建议"、"测试计划"、"测试覆盖"
  - "影响范围"、"变更统计"、"部署注意事项"
  - "问题背景"、"解决方案"、"提交记录"
- ❌ "🤖 Generated with Claude Code" 或任何生成工具标识
- ❌ 文件变更统计（如"39 个文件变更"、"6 files changed"）
- ❌ 提交历史列表
- ❌ 任何形式的 checklist（[ ] 或 [x]）
- ❌ 分隔线（---）
- ❌ 合并目标分支说明

**记住**：用户明确要求只要"主要变更"，其他内容都是噪音。保持简洁，直击要点。

### 6. 输出格式

以清晰的格式呈现给用户，严格按照以下模板：

```
## PR 标题
```
<标题内容>
```

## PR 描述
```markdown
## 主要变更

### 1. <模块名称>
- **<功能点>**：<说明>
  - <细节>

### 2. <模块名称>
- <变更说明>
```
```

**重要提醒**：
- 标题和描述之间不要添加分隔线（---）或空行
- PR 描述部分从 `## 主要变更` 开始，到最后一个变更点结束
- 不要在描述前添加任何引言或概述
- 不要在描述后添加任何总结、建议或标识
- 用户可以直接复制 markdown 代码块中的内容使用

## 边界情况处理

**没有变更**：
如果 `git diff` 显示没有变更，告知用户当前分支与目标分支没有差异，无需创建 PR。

**变更过大**：
如果变更超过 50 个文件或 5000 行代码，建议用户考虑拆分 PR，但仍然生成完整的描述。

**合并冲突**：
如果检测到可能的合并冲突（通过 `git merge-base` 检查），提醒用户先解决冲突。

**未提交的变更**：
如果有未提交的变更（`git status` 显示 modified 文件），提醒用户先提交或暂存。

## 示例

**用户输入**：
> 现在需要提 PR 合并到 test 分支，请生成对应的标题、描述

**输出**：
```
## PR 标题
```
feat: 消息通知系统与市场观点优化
```

## PR 描述
```markdown
## 主要变更

### 1. 消息通知系统
- **新增消息列表查询接口** `GET /api/v1/notification`
  - 支持按状态筛选（pending/completed）
  - 支持按类型筛选（opinion/feedback/order）
  - 支持内容关键词搜索
  - 支持时间排序
  - 分页查询功能
- **数据模型**
  - 新增 Notification 实体及完整 CRUD 操作
  - 包含消息内容、状态、类型、接收者、关联对象等字段
- **完整三层架构实现**
  - Service 层：接口参数转换与业务编排
  - Biz 层：业务逻辑处理
  - Data 层：数据持久化与查询

### 2. 市场观点模块
- 市场观点列表接口增加判空处理
- 市场观点细节功能开发完善
```
```

**注意**：这个示例展示的是完整且正确的输出格式，没有任何多余内容。

## 质量标准

生成的 PR 信息应该：
- ✅ 让审查者 30 秒内理解改动的核心价值
- ✅ 技术负责人能快速评估影响范围
- ✅ 测试人员能识别需要验证的功能点
- ✅ 未来回溯时能快速理解当时做了什么

避免：
- ❌ 过于简单（"更新代码"、"修复 bug"）
- ❌ 过于冗长（超过 50 行描述）
- ❌ 技术细节过载（贴代码片段、列举所有文件）
- ❌ 格式混乱（没有层次、难以阅读）
