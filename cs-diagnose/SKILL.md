# cs-diagnose

## 启动必读

开始任何判断或动作前，先读取 `.codestable/attention.md`；缺失则视为骨架不完整，提示先补齐或运行 `cs-onboard`，不要回退到外部 AI 入口文件。

本技能是 **CodeStable 统一的 bug 诊断入口**，支持两种模式：
- **快速通道**：简单 bug，5 分钟内能定位根因
- **完整诊断**：疑难 bug，需要系统化的 6 阶段诊断

核心理念：**反馈循环是一切的基础**。无论简单还是复杂，都先构建可验证的 pass/fail 信号。

> 共享路径与命名约定看 `.codestable/reference/shared-conventions.md` 第 0 节。

---

## 启动检查与模式判断

1. **确认是 bug 不是新功能需求**——描述"想加 X 功能"的告诉他走 `cs-feat`
2. **看有没有相关 issue 目录**——Glob `.codestable/issues/`，有同类问题先和用户确认是新建还是继续
3. **快速通道判定（唯一正式判定点）**——按用户线索**读一下相关代码**（Grep / Read 定位）：
   - **能一眼确定根因**（能给出 `{文件}:{行号}`、修复改动小 1-2 处、无跨模块影响风险）→ 告诉用户"我已看到问题所在，可以走快速通道：快速构建测试循环、直接修复、简化文档。确认后立即开始"
   - **不能**（根因有多个候选 / 不确定 / 需要更多复现信息 / 性能问题 / 竞态条件）→ 走完整 6 阶段诊断
4. **确定 issue 目录名**——跟用户商定 slug，日期前缀用今天（环境信息 `currentDate`）

## 快速通道（简单 bug）

### 触发条件

同时满足：
1. 5 分钟内读代码能确定根因（明确指出 file:line + 原因）
2. 修复改动很小（1-2 处）
3. 无跨模块影响风险

### 简化流程

**1. 问题记录（对话式，不写独立 report）**

通过对话收集：
- 问题现象（具体异常表现）
- 复现步骤（可执行步骤）
- 期望 vs 实际行为
- 环境信息（模块、相关文件）

**2. 快速反馈循环**

构建最简单的验证方式（优先级）：
1. 现有测试能覆盖 → 直接跑测试
2. 写一个失败测试（< 20 行）
3. HTTP/CLI 脚本（< 10 行）
4. 手动验证步骤（最后手段）

**3. 确认根因并修复**

- 明确陈述根因："`{文件}:{行号}` 的 {具体代码} 存在 {问题描述}"
- 给修复方案（一两句话）
- **等用户确认"对，就这样改"才动手**
- 读 `.codestable/attention.md`
- 按方案定点修复

**4. 验证**

- 反馈循环通过（测试 pass / 脚本 pass / 手动确认）
- 相关测试通过
- 前端改动需浏览器验证（如适用）

**5. 写简化版 diagnosis.md**

模板见 reference.md "快速通道模板"，只记录：
- 问题概述（1-2 段）
- 根因（file:line + 简述）
- 修复方案（改了什么）
- 验证结果

**产物**：`{slug}-diagnosis.md`（简化版）+ `{slug}-fix-note.md`

## 完整诊断（疑难 bug）

### 触发场景

- 根因不明、有多个候选假设
- 问题不稳定复现、概率性出现
- 性能回归、需要性能剖析
- 需要构建专门的复现环境
- 需要仪器化调试（日志、追踪、profiler）

### 六阶段流程

**Phase 1 — 构建反馈循环**（核心）

## 文件放哪儿

```
.codestable/issues/{YYYY-MM-DD}-{slug}/
├── {slug}-diagnosis.md           ← 诊断全记录（必出产物）
├── {slug}-feedback-loop.sh       ← 反馈循环脚本（如有）
├── {slug}-hypotheses.md          ← 假设清单（如有）
└── {slug}-fix-note.md            ← 修复记录（必出产物）
```

日期取**发现问题当天**。slug 能反映问题类型（`auth-race-condition`、`query-perf-regression`）。

`{slug}-diagnosis.md` 记录完整诊断过程：反馈循环构建 → 假设生成 → 验证结果 → 根因确认。这是核心产物，比 cs-issue 的 report+analysis 更完整。

所有文档带 YAML frontmatter（`doc_type=diagnosis` / `doc_type=issue-fix`），便于 `search-yaml.py` 检索。

---

**Phase 1 — 构建反馈循环**（核心）

花充足时间在这里，**拒绝在没有循环的情况下进入假设阶段**。

目标：构建**快速（<30 秒）、确定性、agent 可运行**的 pass/fail 信号。

10 种构建方法（优先级从高到低）：
1. 失败测试（单元/集成/e2e）
2. HTTP 脚本（curl/httpie）
3. CLI 调用 + diff 快照
4. 无头浏览器（Playwright/Puppeteer）
5. 追踪重放（保存真实请求到磁盘）
6. 一次性 harness（最小系统子集）
7. 属性/模糊循环（1000 次随机输入）
8. 二分 harness（git bisect run）
9. 差异循环（旧版 vs 新版）
10. **HITL 脚本**——人工必须参与时，用 `scripts/hitl-loop.template.sh` 模板

循环迭代：能更快吗？信号更清晰吗？更确定吗？

非确定性 bug：目标是**提高复现率到 >50%**（循环 100 次、并行化、添加压力）。

无法构建循环：**停下来明确告知用户**，请求环境访问/artifact/生产仪器化许可。

详见 reference.md "Phase 1 详细指南"。

**Phase 2 — 复现确认**

运行循环，确认：
- [ ] 失败模式与用户描述一致
- [ ] 可跨多次运行复现（或复现率 ≥50%）
- [ ] 精确症状已捕获

不通过则回 Phase 1 改进循环。

**Phase 3 — 生成假设**

生成 **3-5 个排序假设**，每个必须**可证伪**：

> 假设 N：如果 X 是根因，那么改变 Y 会让 bug 消失/改变 Z 会让它更糟。

**给用户看排序列表再测试**。用户可能有领域知识立即重排或排除。

**Phase 4 — 仪器化验证**

每个探针映射到具体假设预测。**每次改变一个变量**。

工具优先级：Debugger > 定向日志 > 永不"全记日志再 grep"。

**标记每条调试日志**：`[DEBUG-{tag}]`，清理时一个 grep 搞定。

性能问题：建立基线测量 → 二分法定位 → 先测量后修复。

**Phase 5 — 修复 + 回归测试**

先写回归测试（如有正确层次）再写修复。无正确层次时，记录"架构阻止 bug 被锁定"。

**Phase 6 — 清理 + 复盘**

完成前必检：
- [ ] 原始复现不再复现
- [ ] 回归测试通过（或无层次已记录）
- [ ] 所有 `[DEBUG-...]` 已移除
- [ ] 一次性原型已删除
- [ ] 修复消息已包含根因说明

**然后问：什么能预防这个 bug？** 架构改变需求交给后续改进。

详见 reference.md "完整诊断模板"。

## 退出条件

**快速通道**：
- [ ] 根因已确认并修复
- [ ] 反馈循环/验证通过
- [ ] `{slug}-diagnosis.md` 已建（简化版）
- [ ] `{slug}-fix-note.md` 已建
- [ ] 用户确认修复完成

**完整诊断**：
- [ ] 所有 6 个 Phase 完成
- [ ] `{slug}-diagnosis.md` 已建（完整版）
- [ ] `{slug}-fix-note.md` 已建
- [ ] 所有仪器化代码已移除
- [ ] 用户确认修复完成

## 收尾提交

按 `shared-conventions.md` 第 4 节"scoped-commit"规则：

**提交范围**：修复代码 + diagnosis.md + fix-note.md + 反馈循环脚本（如有）+ 回归测试（如有）

**Commit 消息**：
```
fix({模块}): {问题一句话}

根因：{简述根因}
修复：{简述修复方案}

诊断记录：.codestable/issues/{issue 目录}/

Co-Authored-By: Claude <noreply@anthropic.com>
```

---

## 退出后推荐

告诉用户："诊断修复完成，完整记录已存档。"

按顺序询问（用户说"不用"立即跳过）：

1. **沉淀技巧**：诊断中用的技术技巧值得复用？→ "记录到 cs-trick？"
2. **沉淀经验**：踩的坑值得警示？→ "记录到 cs-learn？"
3. **架构改进**：暴露了架构问题？→ "提交 improve-codebase-architecture？"
4. **更新架构文档**：影响了架构理解？→ "更新 cs-arch？"
5. **代为提交**：同意时按收尾提交规则执行

## 容易踩的坑

- **没有反馈循环就开始假设**——Phase 1 是基础，跳不得
- **快速通道没确认根因就动手**——必须等用户说"对，就这样改"
- **反馈循环太慢（>2 分钟）**——持续优化到 <30 秒
- **单一假设生成**——必须 3-5 个排序假设
- **仪器化不标记**——清理时找不全，留下调试代码
- **性能问题用日志调试**——应该用 profiler + 基线测量
- **没写回归测试**——bug 会回来
- **Phase 间无 checkpoint**——方向错了才发现，返工成本高
- **快速通道写完整版 diagnosis.md**——浪费时间，用简化版
